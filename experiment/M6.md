# M6 — verifier-reward fine-tuning (the meta / dynamic fine-tune)

> Design + honest limits. Consumers: the paper (§5.5), the trader agent
> (trader.localharness.xyz — it wants a SMALL on-device generator), and, in
> aspiration, any kit language. This file is the germline for the milestone;
> the code that exists today is the reward oracle in `experiment/src/reward.rs`
> and the `s5 reward` CLI.

## The claim, and what it is NOT

Fine-tune a small open-weights model to generate a purpose-sized language by
using the kit's mechanical verifier as the reward — **instead of** distilling a
big teacher onto all-of-Rust and rewarding against rustc. This is the
tempo-x402 result (0.5B model, pass@1 1.5%→16.4%, "compiler-verified
self-play") generalized off rustc onto the kit.

§5 and M6 are the same experiment with one knob moved: **§5 selects from a
frozen model; M6 puts the model in the loop.** Same verifier, same candles,
same reward shape.

**Why a purpose-sized language is the reward, not rustc:**

- `verify()` is FUEL-BOUNDED, so computing a rollout's reward is itself
  guaranteed to terminate. The oracle can never hang the training loop, however
  hostile the program. Reward a model against rustc-plus-run and one generated
  infinite loop stalls the loop; here it cannot exist.
- TOTALITY closes a reward-hacking door: you cannot farm reward by emitting a
  program that loops forever, because nothing does.
- The reward is CPU-cheap and deterministic — thousands of rollouts/second,
  and the same rollouts always score the same, so a checkpoint's rewards
  reproduce from a command even though the model never can.

## The pipeline

Three stages, each a pure function of committed artifacts plus one
nondeterministic generator.

1. **Cold-start (once).** Take the §5 `raw.jsonl`, keep verifier-survivors, SFT
   the base small model on them → checkpoint `C0`. (Position on distillation:
   the DEFAULT cold-start is the base model's OWN temperature-sampled
   survivors; the §5 Opus batch is demoted to an ABLATION the from-base run
   must measurably surpass. That keeps the anti-distillation claim honest — the
   teacher is a baseline to beat, not a crutch.)
2. **Self-play round `r` (repeat ~5–20×).** `C_{r-1}` samples K programs per
   `STYLES` prompt → `s5 reward` batch-scores them → keep the **Ok-rung**
   survivors (see the gate-rung hazard below) that ALSO clear a jittered gate
   on a rolling VALIDATION window, deduped by `nkey` → SFT → `C_r`. Every round
   logs the Reject histogram + the fuel distribution + the distinct-`nkey`
   count — the numbers `score.rs`/`reward.rs` already emit ARE the learning
   curve.
3. **Eval (once).** Generate a fixed N from base, `C0`, each `C_r`, and a
   FRESH, unselected Opus pool, on a held-out TEST window, and report the
   metric below.

**The Rust/Python boundary is exactly two things:** the `s5 reward` CLI
(candidate JSONL in → reward JSONL out) and the generation format (the same
`{source}` shape §5 already uses). Rust owns verify, the reward scalar,
`nkey`, the histogram, fuel, backtest — all cargo-testable now. Python owns
sampling, the SFT/GRPO weight update, checkpoints — none runnable here (no
GPU). The trainer is writable but CI-linted, never GPU-run in CI.

## The reward (shipped: `reward.rs`)

A four-rung validity ladder built directly from `Reject`: compile `0` → run
`1/3` → gate `2/3` → valid strategy `1`. **Train PnL is not in it.** Two
reasons, both anti-hacking:

- **Circularity.** The headline metric is a held-out number; if train PnL is
  the reward, the comparison measures the thing you optimized. Reward must ≠
  eval metric.
- **Curve-fitting.** Continuous train PnL is the single most overfittable
  signal on a few months of one asset.

The facts (PnL, fuel, hash, `nkey`) are EMITTED so a trainer can reshape and
own that choice; the default rewards VALIDITY and leaves generalization for the
experiment to test. `value == ladder(class)` is a tested invariant — dedup and
novelty are applied SEPARATELY (in the trainer's keep-filter), never
multiplied into the value.

### Yield vs edge — the load-bearing sentence

M6 raises **yield**: the held-out survivor rate and behavioral diversity of the
generated pool. It does NOT chase **edge**: finding the profitable strategy
stays the job of §5's unchanged `pick_verified → score_heldout`. This is the
mechanical (not rhetorical) answer to the strongest attack — "validity ≠ edge,
you built a well-formed-noise generator." Correct: M6 is a better *generator*;
§5 is the *selector*; they compose.

## Algorithm

**MVP: rejection-sampling SFT / expert-iteration** — the tempo-x402 recipe,
zero RL infra: sample K, keep the survivors above threshold, SFT, repeat. A
mechanical verifier gives exactly the cheap boolean FILTER this needs — no
reward model, no PPO value network, no KL controller. **Escalation (only on a
pre-registered plateau): GRPO**, driven by the SAME reward CLI unchanged.

Honest concession: **the graded ladder is unused by the MVP.** Expert-iteration
trains only on the kept set via a boolean threshold; the `1/3` and `2/3` rungs
never touch the SFT loss. The ladder earns its place two other ways — as the
histogram-that-is-the-learning-curve, and as the graded signal GRPO needs — not
as MVP reward shaping. A boolean survivor flag suffices for the MVP, and we say
so.

## Reward hacking — the games and the guards

1. **Trivial-survivor mode collapse.** The model finds one minimal program that
   clears the gate and emits near-copies. Guard: `nkey` source-canonical dedup
   (shipped, tested) — caps any key's share of the SFT set. Works below the Ok
   rung, where `equity_hash` is 0 for everything and blind.
2. **The 2/3 gate rung is a farmable local optimum.** `lookback 4;` or a
   comment-only body compiles, runs clean, gate-fails → `2/3` for the absence
   of a strategy; the empty-guard only catches whitespace. Guard: the trainer's
   SFT admission keeps **Ok-rung survivors only**, never gate-rung — the `2/3`
   is a diagnostic/GRPO signal, not an admission threshold.
3. **Gate-threshold gaming.** The model learns the exact 4-trades/16-bars
   thresholds. Guard: jitter `min_trades`/`min_bars` and `Costs` across rounds;
   "hit the threshold" becomes "be robustly active." Eval uses a strictly
   TIGHTER frozen gate than any training round.
4. **`equity_hash` diversity is gameable.** Perturb one constant by a tick →
   new curve → new hash → counted as distinct. Guard: use `nkey`, not
   `equity_hash`, for both the training dedup and the diversity metric. (Honest
   limit: `nkey` catches comment/format clones, not constant-perturbation or
   semantic clones — AST canonicalization is the future refinement.)
5. **Collapse to no-signal.** A no-signal program is gate-rejected (`2/3` <
   `1`), so the ladder disincentivizes it; under GRPO, guard the local optimum
   with an entropy floor + KL to the frozen `C0` reference.

## Eval — and proving the benchmark is non-trivial FIRST

The trap all three judges caught: **held-out survivor rate is not a clean
generalization signal, because the compile rung is DATA-INDEPENDENT.** Whether
a program parses (and largely whether it halts) does not depend on the candles
— a program that compiles on train compiles identically on test. So a raw
"held-out survivor lift" can be almost entirely a compile-rate lift that shows
identically on train: grammar-learning dressed as generalization ("you taught
it to close its brackets").

Two mandatory consequences:

- **Prove the benchmark has out-of-sample teeth before claiming a lift on it.**
  Measure the base model's train-vs-held-out survivor GAP first. If it is ~0,
  the held-out regime does not induce inactivity and the eval axis is INVALID
  until a regime that does is chosen (e.g. a ranging window where a
  trend-fitted active strategy goes silent).
- **The metric is CONDITIONAL, not raw.** Among programs that COMPILE, report
  the held-out GATE-clear rate (genuinely data-dependent) and the train-vs-test
  GAP on that number — plus a **per-style-stratified** rate, because aggregate
  rate can rise by collapsing to the single easiest style.

Other eval discipline: a hard chronological EMBARGO between the last training
window and the test window (iterated per-round resampling otherwise creeps
test-adjacent); a **pre-registered eval gate** (survivor rate is a pure
function of where the gate sits — fix it before seeing results); and a FRESH,
unselected Opus pool on the identical held-out candles (the §5 pool was
selected under §5's windows — reusing it is apples-to-oranges and risks leak).

**Win condition:** the fine-tuned model's conditional held-out gate-clear rate
rises over base and closes a stated fraction of the gap to fresh Opus.
**Null (pre-registered as the EXPECTED PnL result):** held-out PnL CIs straddle
zero. Under a validity-only reward that is not a failure — it is the yield/edge
split working — so a null PnL falsifies nothing and cannot be spun. **What
falsifies the thesis:** no conditional gate-clear lift over base, OR a lift that
is entirely compile-rate (no gap on the conditional metric).

## The "meta" claim — real vs aspiration

Real today: **interface reuse.** The reward is `verify()`, uniform across kit
languages; the trainer takes `(prompt card, reward CLI)` as parameters. Swap
stratlite for another language and the harness shape is unchanged.

Aspiration, named as such: **weight-level meta-learning** (train across
languages, few-shot-adapt to a new one) has no evidence here. And the obvious
N=2 control — `lift(stratlite)` vs `lift(prooflite)` — is NOT free: prooflite
ships no input→expected-output task distribution and no behavioral reward, so
that arm requires authoring a prooflite task corpus first. Scoped as future
work, per GENESIS's "negative/overreach results get named."

## Compute — honest, and why small is the point

~8 styles × K≈128 × ~10–20 iterations at 0.5–1.5B, context <1K tokens, reward
CPU-free: **tens of GPU-hours on a single node** — no datacenter ask.
Rejection-sampling's discard-everything inefficiency is the price of needing no
RL infra. "Small" is not a limitation to apologize for: it is the feature the
trader consumer needs — a generator it can run on-device.

## Layout / the seam

The trainer lives OUTSIDE the kit's workspace (it takes torch; rules 1+5 keep
deps out of the graph and wasm32 green), like `experiment/`. Its own cap
counter in `caps.sh` and its own CI lint job (ruff/mypy, never GPU-run). The
Rust reward core is in `experiment/` and is cargo-tested now; the Python
trainer is the next increment — writable, not runnable here.
